## 역할
inbox에 저장된 원문 요청을 분석해서 구현 가능한 형태로 정제한다.

## 실행

$ARGUMENTS 가 파일 경로이면 그 파일을 읽는다.
없으면: `docs/inbox/` 폴더의 파일 목록을 보여주고 사용자에게 선택하게 한다.

## 분석 수행

원문을 읽고 아래 항목을 분석한다:

**1. 문제 정의**
- 이 요청이 해결하려는 핵심 문제가 무엇인가?
- 유저가 진짜 원하는 것은 무엇인가? (표면적 요청 vs 실제 니즈)

**2. 범위 결정**
- MVP 범위로 줄이면 어떻게 되는가?
- 이번에 반드시 해야 할 것 vs 나중에 해도 되는 것 분리
- 이번 버전의 Out of Scope 명시

**3. 구현 방향**
- 기술적으로 어떻게 구현할 것인가?
- 기존 코드에서 영향받는 부분은?
- Supabase, 인증, 스토리지 등 인프라 영향은?

**4. 리스크**
- 구현 난이도나 기술적 리스크가 있는가?
- 기존 기능에 side effect가 있을 수 있는가?
- 이번 버전에 포함시키기에 너무 크지는 않은가?

## 분석 결과 출력

분석 결과를 명확하게 출력한다.
너무 크거나 리스크가 높으면 분리 또는 보류를 권고한다.

## 사용자 확인 (user_confirm)

분석 결과 출력 후 반드시 확인을 받는다:

```
─────────────────────────────────
분석이 완료됐습니다.

[Y] 태스크 분리 단계로 넘어갑니다 → /wf-planning
[N] 요건을 다시 수집합니다 → /wf-intake
─────────────────────────────────
```

## 완료 처리

사용자가 Y를 선택하면:
1. inbox 파일의 `- [ ] 분석 완료` 체크박스를 체크한다
2. 다음 단계: `/wf-planning` 안내
