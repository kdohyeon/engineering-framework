description: "[Workflow 긴급] 핫픽스 — 프로덕션 장애 긴급 대응 (planning/backlog 단계 스킵)"
---

## 역할
프로덕션에서 긴급 장애가 발생했을 때 빠르게 대응한다.
일반 워크플로우의 planning → backlog → release_plan 단계를 건너뛴다.

사용법: `/wf-hotfix "결제 API 오류로 결제가 안 됨"`

## 실행

$ARGUMENTS 가 있으면 장애 내용으로 사용한다.
없으면 사용자에게 장애 증상을 질문한다.

## 1단계: 긴급 분석

장애 내용을 바탕으로 즉시 분석한다:
- 영향 범위: 어떤 기능이 영향받는가?
- 원인 추론: 가장 가능성 높은 원인은?
- 대응 방향: 가장 빠른 수정 방법은?

```
─────────────────────────────────
긴급 분석 결과:
  증상: [장애 내용]
  영향 범위: [영향받는 기능]
  추정 원인: [원인]
  대응 방향: [수정 방법]

즉시 대응을 시작합니까? (Y/N)
─────────────────────────────────
```

## 2단계: GitHub Issue 생성 (hotfix 레이블)

```bash
gh issue create \
  --title "[HOTFIX] [장애 요약]" \
  --label "hotfix" \
  --body "## 증상
[장애 내용]

## 영향 범위
[영향받는 기능]

## 대응 방향
[수정 방법]"
```

## 3단계: hotfix 브랜치 생성 (main 기준)

```bash
git checkout main
git pull origin main
git checkout -b hotfix/[이슈번호]-[장애-slug]
```

예: `hotfix/42-payment-api-error`

## 4단계: 수정 안내

이슈의 완료 조건을 체크리스트로 제공하고 개발을 안내한다.

## 5단계: 빠른 검증

hotfix는 develop 테스트를 생략하고 직접 main으로 PR을 생성한다.
단, 수정 내용은 로컬에서 반드시 확인한다.

```
─────────────────────────────────
수정이 완료됐습니까?
로컬에서 직접 테스트했습니까? (Y/N)
─────────────────────────────────
```

## 6단계: main으로 직행 PR

```bash
gh pr create \
  --base main \
  --head hotfix/[브랜치명] \
  --title "[HOTFIX] #[번호] [장애 요약]" \
  --body "Closes #[번호]

## 긴급 수정 내용
[수정 내용]

## 검증
- [ ] 로컬 테스트 완료"
```

## 7단계: 패치 버전 릴리즈

PR merge 후 즉시 패치 버전을 릴리즈한다:

```bash
# 현재 버전에서 patch 버전업
# v1.0.2 → v1.0.3
git tag -a v[패치버전] -m "v[패치버전] - [장애 요약] 긴급 수정"
git push origin v[패치버전]

gh release create v[패치버전] \
  --title "v[패치버전] - [장애 요약] 긴급 수정" \
  --notes "[장애 내용] 긴급 패치"
```

## 완료 후

```
─────────────────────────────────
핫픽스 v[버전] 배포 완료 ✓

이제 모니터링을 강화하세요: /wf-monitor
─────────────────────────────────
```
