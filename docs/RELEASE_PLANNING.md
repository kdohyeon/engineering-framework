# 릴리즈 계획 (Release Planning)

릴리즈 계획은 **무엇을 언제 배포할지** 결정하는 과정이다.  
기술적 실행 절차는 [RELEASE_MANAGEMENT.md](RELEASE_MANAGEMENT.md)를 참고.

---

## 릴리즈 주기

정기 릴리즈는 **매월 1회**를 기본으로 한다.

```
매월 1일 기준으로 역산:
  D-14 (2주 전) : 릴리즈 계획 수립 (이번 문서의 전 과정 수행)
  D-7  (1주 전) : 이해관계자 사전 공유 (배포 일정 + 포함 내용)
  D-0  (배포일) : 배포 실행
  D+0  (당일)   : 배포 완료 공유 (변경 내용 + 확인 방법)
```

긴급 장애는 이 주기를 따르지 않는다 → [핫픽스 절차](RELEASE_MANAGEMENT.md#핫픽스-릴리즈-절차)

## Step 1. Backlog 검토 (D-14)

### 검토 순서

1. GitHub Issues에서 `Backlog` 상태(Milestone 미할당) 이슈 전체 목록 확인
2. 각 이슈에 아래 기준으로 점수 매기기

### 선정 기준

| 기준 | 질문 |
|------|------|
| 유저 임팩트 | 이 기능/수정이 얼마나 많은 유저에게, 얼마나 자주 영향을 주는가? |
| 비즈니스 가치 | 리텐션, 전환율, 수익에 직접 연결되는가? |
| 구현 복잡도 | 이번 주기(~2주) 안에 안전하게 완료할 수 있는가? |
| 의존성 | 다른 이슈가 먼저 완료되어야 하는가? |
| 기술 부채 | 이걸 안 하면 나중에 더 큰 비용이 드는가? |

### Claude와 상의할 내용

```
이번 릴리즈 후보 이슈들이야:
- #12: [기능 A]
- #18: [기능 B]
- #25: [버그 C]

각 이슈의 구현 복잡도와 리스크를 평가해줘.
이번 주기(2주)에 안전하게 넣을 수 있는 조합을 추천해줘.
```

### 선정 원칙

- 욕심 부리지 않는다. **덜 넣고 잘 배포**하는 게 낫다
- 한 번에 너무 많은 변경이 들어가면 장애 원인 파악이 어려워진다
- 구현 중 막히면 해당 이슈를 다음 릴리즈로 미룬다 (릴리즈 날짜는 밀지 않는다)

## Step 2. 버전 결정 (D-14)

```
v[X].[Y].[Z]
```

| 자리 | 규칙 | 언제 올리는가 |
|------|------|--------------|
| X | 연 단위 auto-increment | 매년 1월 첫 릴리즈 시 X+1, Y는 1로 리셋 |
| Y | 정기 릴리즈마다 +1 | 매월 정기 배포 시 (월 1회 기준) |
| Z | 세부 배포마다 +1 | 핫픽스, 버그픽스 등 정기 릴리즈 외 배포 시 |

**이번 릴리즈 버전 결정:**

```
현재 버전 확인 (git tag 또는 GitHub Releases)
    ↓
정기 릴리즈인가?  →  Y+1, Z=0  (예: v1.3.0 → v1.4.0)
핫픽스인가?       →  Z+1        (예: v1.4.0 → v1.4.1)
새해 첫 릴리즈인가? → X+1, Y=1, Z=0 (예: v1.12.0 → v2.1.0)
```

**결정 후:** GitHub Milestone 생성 (`v1.4.0`), 선정된 이슈에 Milestone 할당

## Step 3. 롤백 전략 수립 (D-14)

배포 전에 "어떻게 되돌릴지"를 미리 정해둔다.

### 롤백 판단 기준

배포 후 아래 중 하나라도 해당되면 즉시 롤백을 검토한다:

```
- 에러율이 배포 전 대비 2배 이상 증가
- 핵심 기능(로그인, 결제, 주요 화면)이 동작하지 않음
- 배포 후 30분 내 심각한 버그 제보가 2건 이상
```

### 롤백 방법

**방법 A: 이전 버전 재배포 (권장)**
```bash
# 이전 태그로 체크아웃 후 재배포
git checkout v1.1.0
# 배포 스크립트 실행
```

**방법 B: Git Revert (코드베이스 기록 유지)**
```bash
# 문제가 된 merge commit을 revert
git revert -m 1 <merge-commit-hash>
git push origin main
# 재배포
```

**방법 C: Feature Flag (기능 단위 비활성화)**
- 배포는 유지하되 문제 기능만 끄는 방식
- 사전에 feature flag가 코드에 구현되어 있어야 함

### DB 마이그레이션이 포함된 경우

DB 스키마 변경이 포함된 릴리즈는 롤백 난이도가 높아진다.

```
릴리즈에 DB 마이그레이션 포함 여부 확인
    ↓
포함된 경우:
  - 마이그레이션 전 DB 스냅샷/백업 확인
  - 롤백 시 down migration 스크립트 준비
  - 롤백 불가 변경(컬럼 삭제 등)은 별도 절차로 처리
```

## Step 4. 모니터링 계획 (D-14)

배포 후 언제까지 어떻게 지켜볼지 미리 정한다.

### 모니터링 기간

```
배포 직후 ~ 30분 : 집중 모니터링 (자리 비우지 않기)
배포 후 24시간   : 주기적 확인 (2-3시간에 한 번)
배포 후 72시간   : 안정화 확인 후 종료
```

### 확인 항목

| 항목 | 확인 방법 | 이상 기준 |
|------|----------|----------|
| 에러 로그 | 서버 로그 / Sentry | 신규 에러 발생 |
| 응답 속도 | APM / 서버 메트릭 | 배포 전 대비 30% 이상 저하 |
| 핵심 기능 | 직접 QA (로그인, 주요 플로우) | 동작 안 함 |
| 유저 피드백 | 앱 리뷰, CS 채널 | 동일 증상 제보 집중 |

### Slack 알림 설정

배포 전에 Slack 알림이 연결되어 있는지 확인한다.  
→ [Slack 연동 가이드](SLACK_INTEGRATION.md)

## Step 5. 이해관계자 커뮤니케이션

### 5-1. 배포 전 사전 공유 (D-7)

배포 1주 전, 관련된 사람들에게 미리 공유한다.

**공유 대상:** 팀원, 고객사, 베타 유저, 관련 파트너 등 (프로젝트에 맞게 조정)

**Slack / 이메일 템플릿:**

```
[배포 예정 안내] v{버전} - {배포 예정일}

안녕하세요. 다음 주 {날짜}에 새 버전 배포가 예정되어 있어 미리 공유합니다.

📦 포함 내용
- {주요 변경 내용 1}
- {주요 변경 내용 2}
- {버그 수정 내용}

⏰ 배포 예정 시간: {날짜} {시간대} (예: 2026-06-01 오전 10:00 KST)
⏱ 예상 소요 시간: {예상 시간} (예: 약 10-20분, 서비스 중단 없음)

문의 사항은 이 채널로 남겨주세요.
```

### 5-2. 배포 완료 공유 (D+0, 배포 당일)

배포가 완료되면 즉시 공유한다.

**Slack / 이메일 템플릿:**

```
[배포 완료] v{버전} 배포가 완료되었습니다 ✅

📦 변경 내용
- {주요 변경 내용 1}: {유저 관점 설명}
- {주요 변경 내용 2}: {유저 관점 설명}
- {버그 수정}: {어떤 문제가 해결되었는지}

🔗 전체 변경 내역: {CHANGELOG 링크 또는 GitHub Release 링크}

이상한 점이 있으면 바로 알려주세요.
```

### 5-3. 장애 발생 시 커뮤니케이션

배포 후 장애가 발생하면 **빠르게, 사실 기반으로** 공유한다.

**장애 발생 즉시 (인지 후 10분 이내):**

```
[장애 발생] {서비스명} 이상 감지

🔴 현재 상황: {어떤 기능에 문제가 있는지}
👥 영향 범위: {어떤 유저가 영향을 받는지, 예: 전체 / 로그인 유저 / 특정 기능 사용자}
🕐 발생 시각: {시각}
🔍 원인 파악 중

계속 업데이트할게요.
```

**복구 완료 시:**

```
[장애 복구 완료] {서비스명} 정상화

✅ 복구 시각: {시각}
⏱ 장애 지속 시간: {시간} (예: 약 23분)
👥 영향 범위: {영향받은 유저/기능}
🔍 원인: {간단한 원인 설명}
🛠 조치 내용: {어떻게 해결했는지}
📋 재발 방지: {향후 어떻게 방지할지}

불편을 드려 죄송합니다.
```

릴리즈 계획 수립 시 (D-14) 이 체크리스트를 완료한다.

```
[ ] 배포 예정 일시 확정

```

## 릴리즈 계획 → 실행 연결

계획이 완료되면 기술적 실행 단계로 넘어간다.

```
RELEASE_PLANNING.md (이 문서)   →   RELEASE_MANAGEMENT.md
계획 단계                            실행 단계
  - 무엇을 넣을지                      - release 브랜치 생성
  - 버전 결정                          - feature 브랜치 개발
  - 롤백 전략                          - 통합 테스트
  - 모니터링 계획                      - main merge + 태그
  - 이해관계자 커뮤니케이션             - GitHub Release 생성
```
